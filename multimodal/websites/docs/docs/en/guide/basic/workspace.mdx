# Workspace

In Agent TARS' local usage scenario, the directory that is currently allowed to operate on the File System is called the Workspace.

## Default Settings

To avoid affecting your local file system, the default value of Workspace is `$pwd/agent-tars-workspace`, which adds a layer of isolation. When you ask Agent TARS what files it can see:

```text
What files you can see?
```

Because it's a temporary directory, it will be empty on first run:

![](/list-files.png)


## Custom Workspace

You can specify using the `--workspace` parameter through the [CLI](./cli.mdx), such as specifying the current directory:

```bash
agent-tars --workspace=./
```

Assuming the current directory is the [UI-TARS-desktop](https://github.com/bytedance/UI-TARS-desktop) repository, you will see the following output:

![](/list-files-custom-workspace.png)


## Manual Configuration Management

For consistent configuration across projects, you can manually manage configuration files using several approaches:

### Project-Specific Configuration

Create configuration files directly in your project directories:

```typescript
// ./tarko.config.ts
import { AgentAppConfig } from '@tarko/interface';

const config: AgentAppConfig = {
  model: {
    provider: 'openai',
    id: 'gpt-4o',
    apiKey: process.env.OPENAI_API_KEY, // Using environment variables is recommended
  },
};

export default config;
```

### Shared Configuration Files

Create a shared configuration file and reference it using the `--config` flag:

```bash
# Create shared config
mkdir -p ~/configs
cat > ~/configs/tarko.config.ts << 'EOF'
import { AgentAppConfig } from '@tarko/interface';

const config: AgentAppConfig = {
  model: {
    provider: 'openai',
    id: 'gpt-4o',
    apiKey: process.env.OPENAI_API_KEY,
  },
};

export default config;
EOF

# Use shared config
tarko --config ~/configs/tarko.config.ts
```

### Environment Variables

Set up environment variables for consistent model provider credentials:

```bash
# Add to your shell profile (.bashrc, .zshrc, etc.)
export OPENAI_API_KEY="your-api-key"
export ANTHROPIC_API_KEY="your-anthropic-key"

# Use environment variables in config
tarko --model.provider openai --model.apiKey $OPENAI_API_KEY
```

### CLI Configuration Overrides

Override specific settings per execution:

```bash
# Override model provider
tarko --model.provider anthropic --model.id claude-3-5-sonnet-20241022

# Override workspace
tarko --workspace ./my-project-workspace

# Combine multiple overrides
tarko --workspace ./data --model.provider openai --model.id gpt-4
```

## FAQ

### Can I have multiple configurations?

Yes! You can create different configuration files for different projects or use cases:

- **Project-specific**: Create `tarko.config.{ts,yaml,json}` in each project directory
- **Shared configs**: Create multiple config files and reference them with `--config`
- **Environment-based**: Use environment variables to switch between different setups

### How do I update my configuration?

Simply edit the configuration file using any text editor. If you're using TypeScript, make sure to install any necessary dependencies first:

```bash
npm install @tarko/interface
```

### Can I share my configuration with others?

Yes! Configuration files can be shared using Git or any other version control system. Just make sure to:

- Use environment variables for sensitive information like API keys
- Document required environment variables in your README
- Consider creating a template config file for others to copy

### How do I specify different API keys or model providers?

You have several options:

1. **Environment variables** (recommended):
```bash
export OPENAI_API_KEY="your-key"
tarko --model.provider openai
```

2. **CLI overrides**:
```bash
tarko --model.provider openai --model.apiKey YOUR_API_KEY
```

3. **Config file**:
```typescript
// tarko.config.ts
export default {
  model: {
    provider: 'openai',
    apiKey: process.env.OPENAI_API_KEY,
  },
};
```

### How do I use configurations in CI/CD pipelines?

For continuous integration environments:

1. **Create project-specific config**: Add `tarko.config.{ts,yaml,json}` to your repository
2. **Use environment variables**: Set API keys and other secrets as CI environment variables
3. **Explicit workspace**: Always specify `--workspace` to control file operations

```bash
# Example CI command
tarko --workspace ./ci-workspace --model.apiKey $CI_OPENAI_API_KEY
```
